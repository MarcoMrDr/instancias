# Sitio web de gestión de eventos (Next.js + Supabase + Stripe + Vercel)

## Roles implementados

- **superusuario**: control administrativo de eventos (registrar eventos, visualizar eventos y comprar).
- **usuario**: solo visualizar eventos y comprar boletos.

## Regla de negocio solicitada

- Cualquier persona que se registre queda como **usuario** por defecto.
- Solamente el correo del propietario (`NEXT_PUBLIC_SUPERUSER_EMAIL`) se marca como **superusuario** al registrarse desde la app.

## Páginas

- **Inicio** (`/`)
- **Login / Registro** (`/login`)
- **Registro de eventos** (`/registro-eventos`) *(solo superusuario)*
- **Visualizar eventos** (`/eventos`)
- **Carrito de compra** (`/carrito`)


## ¿Dónde se guardan los usuarios?

- El usuario autenticado se crea en `auth.users` (schema interno de Supabase Auth).
- El rol se guarda en `public.perfiles`.
- Los usuarios de login se sincronizan en `public.usuarios` automáticamente desde la app.
- En el panel de Supabase, revisa usuarios en **Authentication > Users**.
- Si borraste/recreaste tablas, cierra sesión y vuelve a entrar para regenerar el perfil en `public.perfiles`.
- Si no ves filas en `public.perfiles`, ejecuta `supabase/setup.sql` para crear tabla/trigger/policies.


## ¿Cuándo se guarda en `public.usuarios`?

- Desde este proyecto, se sincroniza automáticamente al registrarse/iniciar sesión.
- Se guarda `email` y un `nombre` base (antes del `@`) si no existe aún.
- Si no se inserta, revisa que hayas ejecutado `supabase/setup.sql` para crear las políticas RLS de `usuarios`.


## Productos extras (manuales por ahora)

- Los extras que no son boletos se definen en `lib/manualProducts.ts`.
- Actualmente ya vienen dos para prueba:
  - `Botella de vino`
  - `Cigarros`
- Se muestran en la misma página de eventos y se agregan al mismo carrito.

## ¿Tengo que crear productos en Stripe?

- **No es obligatorio** para este flujo actual.
- El checkout usa `price_data` dinámico en el backend (`app/api/checkout/route.ts`), por eso puedes agregar productos manuales en el código y se cobran en una sola sesión.
- Más adelante, si quieres inventario/facturación más avanzada en Stripe, sí conviene crear Products/Prices en Stripe y guardar sus `price_id`.

## ¿Cómo vuelvo mi cuenta superusuario?

1. Regístrate normalmente en `/login`.
2. Ejecuta el `UPDATE` del final de `supabase/setup.sql` cambiando el email por el tuyo.
3. Cierra sesión y vuelve a iniciar para recargar el rol.

## 1) Probar en VSCode

1. Abre el proyecto en VSCode.
2. Abre una terminal integrada.
3. Ejecuta:

```bash
npm install
npm run dev
```

4. Abre `http://localhost:3000`.

## 2) Configurar variables de entorno

Crea un archivo `.env.local` en la raíz:

```bash
NEXT_PUBLIC_SUPABASE_URL=tu_url_de_supabase
NEXT_PUBLIC_SUPABASE_ANON_KEY=tu_anon_key
NEXT_PUBLIC_SUPERUSER_EMAIL=tu-correo-superusuario@dominio.com
STRIPE_SECRET_KEY=tu_stripe_secret_key
STRIPE_WEBHOOK_SECRET=tu_webhook_secret
SUPABASE_SERVICE_ROLE_KEY=tu_service_role_key
NEXT_PUBLIC_APP_URL=http://localhost:3000
```


> `SUPABASE_SERVICE_ROLE_KEY` es recomendable para idempotencia (guardar `stripe_sessions_procesadas`). Si no la configuras, el sistema usa fallback con `NEXT_PUBLIC_SUPABASE_ANON_KEY` para intentar descontar, pero sin control robusto de duplicados.

## 3) Configurar Supabase

### SQL sugerido para tablas y roles

> También puedes ejecutar directamente `supabase/setup.sql` (incluye trigger y update para superusuario).

```sql
create table if not exists public.usuarios (
  id bigint generated by default as identity primary key,
  nombre text not null,
  email text not null unique,
  created_at timestamp with time zone default timezone('utc', now())
);

create table if not exists public.eventos (
  id bigint generated by default as identity primary key,
  titulo text not null,
  descripcion text not null,
  fecha date not null,
  precio numeric(10,2) not null check (precio > 0),
  limite_boletos integer not null check (limite_boletos >= 0),
  created_at timestamp with time zone default timezone('utc', now())
);

create table if not exists public.perfiles (
  id uuid primary key references auth.users(id) on delete cascade,
  rol text not null check (rol in ('superusuario', 'usuario')),
  created_at timestamp with time zone default timezone('utc', now())
);
```

### Recomendación robusta (backend)

Para máxima seguridad, crea un trigger en Supabase que inserte automáticamente rol `usuario` en `perfiles` cada vez que se registre un usuario en `auth.users`, y luego cambia manualmente tu cuenta a `superusuario`.

> Si usas solo la lógica frontend, igualmente funciona para iniciar, pero la forma robusta es con trigger y políticas RLS.

### Política recomendada (RLS)

- En `eventos`: `SELECT` para usuarios autenticados.
- En `eventos`: `INSERT/UPDATE/DELETE` solo para rol `superusuario`.
- En `perfiles`: `SELECT` para el propio usuario y para superusuario.


## Error común: `new row violates row-level security policy for table "eventos"`

Si te aparece al crear eventos:

1. Ejecuta nuevamente `supabase/setup.sql` completo (incluye políticas de `eventos`).
2. Asegúrate de que tu cuenta tenga rol `superusuario` en `public.perfiles`.
3. Cierra sesión y vuelve a iniciar para refrescar el rol en la app.


## Restar boletos automáticamente después del pago

Para descontar `limite_boletos` cuando Stripe confirma pago:

1. Usa el endpoint `POST /api/stripe/webhook` (ya incluido en el proyecto).
2. Configura en Stripe Dashboard un webhook al URL:
   - Local: `http://localhost:3000/api/stripe/webhook` (usando Stripe CLI)
   - Producción: `https://tu-dominio.com/api/stripe/webhook`
3. Agrega en entorno:
   - `STRIPE_WEBHOOK_SECRET`
   - `SUPABASE_SERVICE_ROLE_KEY`

El webhook procesa `checkout.session.completed` y descuenta boletos de los ítems tipo `boleto-*`.

> Además del webhook, el frontend confirma la compra al volver de Stripe (`/api/checkout/confirm`) usando `session_id` en la URL de éxito, como respaldo por si el webhook no llega en local.


### Prueba local rápida con Stripe CLI

```bash
stripe listen --forward-to localhost:3000/api/stripe/webhook
```

Ese comando te dará un `whsec_...` que debes poner en `STRIPE_WEBHOOK_SECRET`.


### Validación recomendada en base de datos

Ejecuta después de una compra para verificar que sí se descontó:

```sql
select id, titulo, limite_boletos
from public.eventos
order by id;
```

La sesión de Stripe procesada se guarda en:

```sql
select * from public.stripe_sessions_procesadas order by processed_at desc;
```

> Importante: si ya tenías el proyecto corriendo, vuelve a ejecutar `supabase/setup.sql` para crear la función `descontar_boletos` y la tabla `stripe_sessions_procesadas`.
>
> Si Stripe se queda cargando al terminar el pago, valida `NEXT_PUBLIC_APP_URL`. Esa URL se usa para `success_url/cancel_url`; si falta, el backend de checkout ahora intenta usar automáticamente el `Origin` de la petición.

## 4) Flujo de compra

1. Superusuario crea eventos con precio y límite de boletos.
2. Usuario inicia sesión.
3. Usuario visualiza eventos y agrega boletos al carrito.
4. Usuario paga con Stripe desde `/carrito`.


## Stripe: clave correcta y pruebas sin pago real

Si aparece el error `Invalid API Key provided`:

- En backend (`STRIPE_SECRET_KEY`) usa **solo** `sk_test_...` para pruebas o `sk_live_...` para producción.
- **No** uses `pk_test_...` / `pk_live_...` (publishable) en el backend.
- Después de cambiar `.env.local`, reinicia `npm run dev`.

### Pruebas sin cobro real

1. Activa **Test mode** en Stripe Dashboard.
2. Usa `STRIPE_SECRET_KEY=sk_test_...`.
3. Paga en Checkout con tarjeta de prueba `4242 4242 4242 4242`, fecha futura, CVC cualquiera, ZIP cualquiera.
4. Verifica el intento en Stripe Dashboard > Payments (modo test).
